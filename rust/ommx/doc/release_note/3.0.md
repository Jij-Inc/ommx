# Rust SDK 3.0.x

> **Note:** This page groups all changes leading to the stable 3.0.0 release,
> including alpha, beta, and release-candidate development, by design topic and
> user workflow. After 3.0.0, changes in the 3.0.x line are recorded as
> pull-request-based entries in the final `3.0.x updates` section.

This page covers the Rust SDK 3.0 release line. It starts with the v3 redesign
overview for users upgrading from 2.6.1 and keeps all pre-3.0.0 work organized
by user-facing topic.

The redesign notes are organized by design topic, not by individual pull
request. Related pull requests are listed at the end of each topic as
provenance.

Topic headings use two markers throughout this page:

- **рЯІЃ Modeling and optimization** вАФ mathematical modeling features,
  transformations, and solver-facing semantics.
- **вЪЩпЄП SDK architecture and implementation** вАФ internal data representation,
  normalization, serialization, storage, and supporting infrastructure.

## Redesign overview

The 3.0.0 line is a major revision of the Rust SDK:

### рЯІЃ Modeling and optimization

- **Indicator / one-hot / SOS1** are first-class constraint types alongside
  regular `Constraint`, with their own ID types and collection slots on
  `Instance`.
- An **instance class** is a set of `Instance` values defined by OMMX
  structural conditions. An adapter input is classified by membership in that
  set. If preparation produces another instance to use as input, membership
  is checked on that value.
- Coefficient and function arithmetic now preserves the finite, non-zero
  coefficient invariant through fallible operators, with cancellation
  represented by removing zero terms.

### вЪЩпЄП SDK architecture and implementation

- The resulting constraint-kind √Ч lifecycle combinatorics is tamed by a
  shared [`ConstraintType`](crate::ConstraintType) abstraction and a
  [`Stage`](crate::Stage) type parameter, so each kind is one generic
  struct rather than four hand-written ones.
- Constraints are grouped into generic
  [`ConstraintCollection<T>`](crate::ConstraintCollection) /
  [`EvaluatedCollection<T>`](crate::EvaluatedCollection) /
  [`SampledCollection<T>`](crate::SampledCollection) wrappers on
  `Instance`, `ParametricInstance`, `Solution`, and `SampleSet`;
  because the `id` field is gone from individual constraints, the
  **host is the natural unit of serialization**
  (`Instance::to_v1_bytes` / `to_v2_bytes`,
  `ParametricInstance::to_v1_bytes` / `to_v2_bytes`,
  `Solution::to_v1_bytes` / `to_v2_bytes`, and
  `SampleSet::to_v1_bytes` / `to_v2_bytes`).
- The Rust SDK introduces normalized **`ommx.v2` protobuf roots** for
  `Instance`, `ParametricInstance`, `Solution`, and `SampleSet`. These roots
  serialize first-class special constraints, table-owned IDs, labels/context,
  and `required_features` directly, while the legacy `ommx.v1` path remains
  available through explicit `to_v1_bytes` / `from_v1_bytes` methods.
- The companion
  [`ommx-pyo3-bridge`](https://docs.rs/ommx-pyo3-bridge) crate gives
  independently built PyO3 modules typed return wrappers for canonical Python
  OMMX classes without passing Rust or PyO3 wrapper objects across a
  shared-library boundary.
- Modeling labels (`name`, `subscripts`, `parameters`, `description`) move
  off each constraint, decision variable, and named function into
  per-collection **Struct-of-Arrays label/context stores**. Constraint
  `provenance` is kept in `ConstraintContext`, separate from the label.
  These stores are queried through narrow per-host accessors
  (`instance.constraint_context()`, `instance.variable_labels()`,
  `instance.named_function_labels()`, вА¶). One canonical store per
  collection, two views on top: per-id wrapper getters for one-off
  reads and `*_df` for bulk analysis.
- Decision variables, parameters, and named functions now follow the same table ownership rule:
  [`DecisionVariable`](crate::DecisionVariable) is row data containing
  only `kind` and `bound`; the [`VariableID`](crate::VariableID),
  modeling labels, and fixed values live on
  lifecycle-stage-parameterized [`DecisionVariableTable`](crate::DecisionVariableTable)
  for `Instance` / `ParametricInstance`, while
  [`EvaluatedDecisionVariableTable`](crate::EvaluatedDecisionVariableTable)
  and [`SampledDecisionVariableTable`](crate::SampledDecisionVariableTable)
  are aliases for the evaluated and sampled stages on `Solution` / `SampleSet`.
  [`ParametricInstance`](crate::ParametricInstance) stores parameter IDs
  and labels in [`ParameterTable`](crate::ParameterTable), while concrete
  parameter values remain inputs to
  [`ParametricInstance::with_parameters`](crate::ParametricInstance::with_parameters).
  [`NamedFunction`](crate::NamedFunction) likewise stores only the
  [`Function`](crate::Function); [`NamedFunctionID`](crate::NamedFunctionID)
  lives on the enclosing named-function maps.
- The v3 **OMMX Artifact** API is registry-first: mutable
  [`ArtifactDraft`](crate::artifact::ArtifactDraft)s commit to immutable
  [`LocalArtifact`](crate::artifact::LocalArtifact)s represented as OCI Image
  Manifests, while `.ommx` archives and remote registries are exchange and
  distribution boundaries around the Local Registry.
- A new **Experiment** session model records Runs, solves, samplings,
  attachments, parameters, traces, and diagnostics, then commits or
  checkpoints them as OMMX Artifacts with explicit recovery and lineage.
- The default **error surface** is a single
  [`ommx::Result<T>`](crate::Result) (re-export of `anyhow::Result`)
  with diagnostic context emitted through `tracing` from the new
  fail-site macros. A handful of typed signal types
  (`BoundError`, `DecisionVariableError`, `DuplicatedSampleIDError`,
  `SubstitutionError`, вА¶) stay typed at their public-API entry points
  for callers that recover by discriminant.
- The long-running migration away from the proto-generated `v1_ext`
  helpers finishes: domain types (`Instance`, `Constraint`,
  `DecisionVariable`, вА¶) are the primary API, and `v1::*` is reserved for
  wire-format interop.

See the [migration guide](crate::doc::migration_guide) for the detailed
v2 вЖТ v3 upgrade path. This page is a topic-oriented summary of what changed and
why.

## рЯІЃ First-class special constraint types

Special-structure constraints are now first-class domain objects, parallel to
the regular [`Constraint`](crate::Constraint) rather than hints hanging off it:

- [`IndicatorConstraint`](crate::IndicatorConstraint) вАФ encoding
  `indicator_variable = 1 вЖТ f(x) {=,вЙ§} 0`. **New in v3.**
- [`OneHotConstraint`](crate::OneHotConstraint) вАФ exactly one of a set of
  binary variables is 1. Previously expressed as a
  `ConstraintHints::OneHot` hint on a regular equality constraint; now a
  constraint type in its own right.
- [`Sos1Constraint`](crate::Sos1Constraint) вАФ at most one of a set of
  variables is non-zero. Previously `ConstraintHints::Sos1`; now first-class.

Each kind has its own ID type
([`IndicatorConstraintID`](crate::IndicatorConstraintID),
[`OneHotConstraintID`](crate::OneHotConstraintID),
[`Sos1ConstraintID`](crate::Sos1ConstraintID)) to prevent accidental
cross-type lookups, and lives in its own collection slot on
[`Instance`](crate::Instance).

Related PRs: [#790](https://github.com/Jij-Inc/ommx/pull/790),
[#798](https://github.com/Jij-Inc/ommx/pull/798).

## вЪЩпЄП Stage parameter and the `ConstraintType` trait

Promoting indicator / one-hot / SOS1 to first-class types alongside
regular `Constraint` would have multiplied the number of concrete
constraint structs by the number of lifecycle states each kind can be
in (created / evaluated / sampled, with removal handled separately).
Hand-writing the resulting 4 √Ч 3 = 12 concrete struct definitions
wasn't going to scale.

The core refactor of 3.0 collapses the matrix: every constraint kind
is one generic struct parameterized by a [`Stage`](crate::Stage)
marker, and the [`ConstraintType`](crate::ConstraintType) trait names
the concrete `Created` / `Evaluated` / `Sampled` types per kind so
generic code (iteration, feasibility checks, DataFrame rendering,
adapter conversion) is written once and applied uniformly. The
[`EvaluatedConstraintBehavior`](crate::EvaluatedConstraintBehavior) /
[`SampledConstraintBehavior`](crate::SampledConstraintBehavior) traits
expose the per-kind feasibility surface in the same style.

Two knock-on simplifications:

- **No `Removed` stage.** Removal is collection-level state вАФ see the
  next section.
- **No `id` field on the struct.** A constraint's ID lives on the
  enclosing `BTreeMap` key, which was already the single source of
  truth.

The migration guide's [Constraint Field Access](crate::doc::migration_guide#1-constraint-field-access)
and [New Types](crate::doc::migration_guide#new-types) sections cover
the struct shapes and the trait family in full.

Related PRs: [#789](https://github.com/Jij-Inc/ommx/pull/789),
[#795](https://github.com/Jij-Inc/ommx/pull/795),
[#796](https://github.com/Jij-Inc/ommx/pull/796),
[#806](https://github.com/Jij-Inc/ommx/pull/806).

## вЪЩпЄП Collections and serialization

A trio of generic collection wrappers holds constraints uniformly
across every kind and every stage:
[`ConstraintCollection<T>`](crate::ConstraintCollection) on
`Instance` / `ParametricInstance` (active + removed paired with a
[`RemovedReason`](crate::RemovedReason)),
[`EvaluatedCollection<T>`](crate::EvaluatedCollection) on
[`Solution`](crate::Solution) (evaluated + removed-reason map), and
[`SampledCollection<T>`](crate::SampledCollection) on
[`SampleSet`](crate::SampleSet) (per-sample variants). Iteration,
feasibility checks, and `RemovedReason` handling work the same way
across every constraint kind at every stage, so adapter / Solution /
SampleSet code doesn't need to special-case the four kinds.

**Serialization moves to the host level.** With the `id` field gone
from individual constraints and labels/context living in a per-collection
SoA store (next section), a single element can no longer round-trip
on its own. Per-element `to_bytes` / `from_bytes` are not provided on
any constraint kind or its evaluated / sampled counterpart; use
`Instance::to_v1_bytes` / `from_v1_bytes` or `to_v2_bytes` /
`from_v2_bytes`, and the corresponding methods on
`ParametricInstance`, `Solution`, and `SampleSet`, as the entry points.
Each host-level root encodes every constraint kind together with IDs and
labels/context in one protobuf root.

The migration guide's [ConstraintCollection](crate::doc::migration_guide#constraintcollection)
and [EvaluatedCollection / SampledCollection](crate::doc::migration_guide#evaluatedcollection--sampledcollection)
reference cards list the public methods on each.

Related PRs: [#789](https://github.com/Jij-Inc/ommx/pull/789),
[#795](https://github.com/Jij-Inc/ommx/pull/795),
[#806](https://github.com/Jij-Inc/ommx/pull/806),
[#821](https://github.com/Jij-Inc/ommx/pull/821).

## вЪЩпЄП `ommx.v2` serialization

The Rust SDK can now serialize and deserialize normalized top-level root objects
through the new `ommx.v2` protobuf roots. Use
[`Instance::to_v2_bytes`](crate::Instance::to_v2_bytes),
[`ParametricInstance::to_v2_bytes`](crate::ParametricInstance::to_v2_bytes),
[`Solution::to_v2_bytes`](crate::Solution::to_v2_bytes), or
[`SampleSet::to_v2_bytes`](crate::SampleSet::to_v2_bytes) to write v2 bytes,
and the corresponding `from_v2_bytes` methods to read them.

The v2 writer preserves the normalized ownership model: table keys own IDs,
modeling labels and fixed values serialize as sidecar columns, constraint
contexts carry modeling labels plus provenance, and active/removed constraint
membership stays on the constraint collection. First-class indicator, one-hot,
and SOS1 collections are serialized directly and top-level roots populate
`required_features` so older readers can reject unsupported semantic features
instead of silently interpreting a weaker model.

The legacy `ommx.v1` path remains available through explicit
`to_v1_bytes` / `from_v1_bytes` names. Because v1 has no first-class
representation for indicator, one-hot, or SOS1 constraints,
`Instance::to_v1_bytes` and `ParametricInstance::to_v1_bytes` return
`Err` instead of panicking when those collections are non-empty; use v2 for
normalized special-constraint payloads. Artifact and Experiment typed
storage now writes v2 payloads for `Instance`, `ParametricInstance`,
`Solution`, and `SampleSet`, while typed readers still accept existing v1
payload layers for backward compatibility.

Related PRs: [#982](https://github.com/Jij-Inc/ommx/pull/982),
[#983](https://github.com/Jij-Inc/ommx/pull/983),
[#984](https://github.com/Jij-Inc/ommx/pull/984),
[#989](https://github.com/Jij-Inc/ommx/pull/989).

## вЪЩпЄП Modeling labels and constraint context on the enclosing collection

Constraints, decision variables, and named functions used to carry their
labels (`name`, `subscripts`, `parameters`, `description`) inline on each
element. In v3 the same modeling-label fact lives in **one canonical place
per collection**: a Struct-of-Arrays label/context store keyed by ID, riding
alongside the constraint / variable / named-function map. Constraint
`provenance` is part of `ConstraintContext`, not part of `ModelingLabel`.
Per-element structs shrink to their intrinsic data and the SoA store is the
canonical source for both per-id reads and bulk DataFrame analysis.

Three store families share one shape:
[`ConstraintContextStore<ID>`](crate::ConstraintContextStore) on
every constraint-kind collection (the same store rides through
[`EvaluatedCollection<T>`](crate::EvaluatedCollection) and
[`SampledCollection<T>`](crate::SampledCollection) so context is
available at every stage),
[`DecisionVariableTable`](crate::DecisionVariableTable), which owns
decision-variable definition rows, fixed values, and the
[`VariableLabelStore`](crate::VariableLabelStore) together on
`Instance` / `ParametricInstance`,
[`EvaluatedDecisionVariableTable`](crate::EvaluatedDecisionVariableTable) and
[`SampledDecisionVariableTable`](crate::SampledDecisionVariableTable), which
own decision-variable result rows and labels on `Solution` / `SampleSet`, and
[`NamedFunctionLabelStore`](crate::NamedFunctionLabelStore) inside
[`NamedFunctionTable`](crate::NamedFunctionTable), which owns named-function
rows and labels together.

The split lets the type system enforce invariants more tightly.
`ConstraintCollection<T>` no longer exposes raw active/removed/context map
mutation to `Instance` transformation code. Transformations go through
operation-level collection effects such as fresh active insertion with context,
active-row rewrites, lifecycle-preserving replacement, relax, and restore
through a host-supplied normalizer. External callers still go through the
validating `Instance::add_*` / `relax_*` / `restore_*` family, which keep
variable-id validity (every `id` referenced by a constraint exists in
`decision_variables`), active/removed disjointness, and label/provenance
sidecar ownership together.

The remaining raw split escape hatches are also closed: table and collection
owners no longer expose public `into_parts` methods. Host-level protobuf
serialization and root conversions use crate-internal helpers that join rows
with their labels/context or preserve lifecycle membership directly, so callers
cannot accidentally separate rows from sidecars.

The Python side wraps the same SoA store with two parallel
user-facing changes:

- `instance.constraints[id]` and the parallel constraint / variable
  accessors return live `AttachedX` write-through handles.
- `*_df()` methods (with `kind=` / `include=` / `removed=` parameters)
  plus six long-format sidecar DataFrames serve bulk analysis directly
  off the SoA store.

See the [Python SDK v2 to v3 Migration Guide](https://jij-inc-ommx.readthedocs-hosted.com/en/latest/migration/python_sdk_v2_to_v3.html)
¬І9вАУ11 for the user-facing version.

The migration guide's [Modeling labels and constraint context](crate::doc::migration_guide#modeling-labels-and-constraint-context)
section has the per-host accessor list and the store API reference.

## вЪЩпЄП Decision-variable table ownership

The Rust SDK now has explicit decision-variable table owners. The map key owns
the [`VariableID`](crate::VariableID), the row owns only intrinsic data, and the
table owns modeling labels. `Instance` and `ParametricInstance` use
[`DecisionVariableTable`](crate::DecisionVariableTable), which also owns fixed
values; `Solution` uses
[`EvaluatedDecisionVariableTable`](crate::EvaluatedDecisionVariableTable), and
`SampleSet` uses
[`SampledDecisionVariableTable`](crate::SampledDecisionVariableTable).
These are the same table owner parameterized by the shared lifecycle stages used
by constraints, so row IDs and modeling labels are validated by one
implementation while fixed values remain a created-stage column.

This removes the remaining duplicate ID source from the Rust-side row structs.
Construct `DecisionVariable` rows with `DecisionVariable::new(kind, bound, atol)`
or no-argument factories such as `DecisionVariable::binary()`, then insert them
under the desired `VariableID` key. The row still owns the `kind`/`bound`
invariant: safe construction and bound mutation normalize `bound` with the
caller-provided `ATol`. `EvaluatedDecisionVariable::new`
and `SampledDecisionVariable::new` still take the ID as a separate argument so
non-finite value errors can report the table key, but the resulting row data
does not store that ID.
For direct created-stage table construction, use
`DecisionVariableTable::with_fixed_values(entries, labels, fixed_values, atol)`.
An empty `fixed_values` map represents the same table schema with no fixed
rows; there is no separate empty-sidecar constructor.

The deprecated `Solution::new` constructor was removed. It was a safe API that
skipped host-level validation by wrapping `SolutionBuilder::build_unchecked`.
Use `Solution::builder().build()` for validated construction, or call the
unsafe `build_unchecked` path only when the enclosing code has already
guaranteed the `Solution` invariants.

This is part of the normalization work tracked in
[#958](https://github.com/Jij-Inc/ommx/issues/958).

## вЪЩпЄП Instance-owned decision-variable state

Decision-variable state is now owned by [`Instance`](crate::Instance), rather
than by the derived `DecisionVariableAnalysis` view or by fields on individual
rows. Fixed values live in the created-stage decision-variable table,
dependency assignments live on the enclosing Instance, and evaluation routes
through [`Instance::populate_state`](crate::Instance::populate_state).

Related PRs: [#944](https://github.com/Jij-Inc/ommx/pull/944),
[#959](https://github.com/Jij-Inc/ommx/pull/959).

## рЯІЃ Decision-variable roles and partial-evaluation semantics

The old analysis-first role API is replaced by direct owner queries such as
`decision_variable_roles`, `decision_variable_role`,
`fixed_decision_variables`, `dependent_decision_variable_ids`, and
`irrelevant_decision_variable_ids`. `decision_variable_usage` remains the
reverse-usage view for variables referenced by the objective, active
constraints, and named functions.

Partial evaluation preserves the same partition. A dependent-variable value is
treated as a consistency assertion when its dependency can be evaluated; a
dependency reduced to a constant moves into the fixed-value column. Attempts
to fix a dependent key without that normalization are rejected instead of
creating a variable that is simultaneously fixed and dependent.

See the [decision-variable role migration](crate::doc::migration_guide#decision-variable-analysis-helpers)
for replacement patterns.

Related PRs: [#946](https://github.com/Jij-Inc/ommx/pull/946),
[#962](https://github.com/Jij-Inc/ommx/pull/962),
[#973](https://github.com/Jij-Inc/ommx/pull/973).

## вЪЩпЄП Named-function table ownership

Named functions now follow the same table-owned ID model as constraints and
decision variables. The Rust SDK row structs no longer carry their own
[`NamedFunctionID`](crate::NamedFunctionID):

- [`NamedFunction`](crate::NamedFunction) stores only the intrinsic
  [`Function`](crate::Function).
- [`EvaluatedNamedFunction`](crate::EvaluatedNamedFunction) stores the
  evaluated value and used decision-variable IDs.
- [`SampledNamedFunction`](crate::SampledNamedFunction) stores sampled values
  and used decision-variable IDs.

The ID, row map, and modeling labels now live together in
[`NamedFunctionTable`](crate::NamedFunctionTable). `Instance` and
`ParametricInstance` store `NamedFunctionTable<NamedFunction>`, `Solution`
stores `NamedFunctionTable<EvaluatedNamedFunction>`, and `SampleSet` stores
`NamedFunctionTable<SampledNamedFunction>`. Legacy `ommx.v1` protobuf messages
still carry inline IDs; Rust parse drains them into table keys and Rust
serialization fills them from table keys.

Mutable named-function row views are not exposed from host objects. In
particular, [`Instance::new_named_function`](crate::Instance::new_named_function)
now returns the allocated [`NamedFunctionID`](crate::NamedFunctionID) rather
than `&mut NamedFunction`, so callers cannot invalidate a checked `Instance` by
editing the function body after insertion.

Related PR: [#964](https://github.com/Jij-Inc/ommx/pull/964).

## вЪЩпЄП Parameter table ownership

[`ParametricInstance`](crate::ParametricInstance) now stores its parameter
universe as a [`ParameterTable`](crate::ParameterTable) rather than
`BTreeMap<VariableID, v1::Parameter>`. Parameter IDs intentionally stay in the
shared [`VariableID`](crate::VariableID) namespace with decision variables,
because [`Function`](crate::Function) references can only be interpreted as
decision variables or parameters by the enclosing `ParametricInstance`.

`ParameterTable` owns the parameter ID set and
[`ParameterLabelStore`](crate::ParameterLabelStore). It enforces the
table-level invariant that labels cannot reference unknown parameter IDs.
`ParametricInstance` remains responsible for the host-level invariants:
parameter IDs and decision-variable IDs must be disjoint, expressions may only
reference IDs from the combined namespace, and structural decision-variable
positions such as indicator / one-hot / SOS1 members cannot use parameter IDs.

Rust callers should pass `ParameterTable` to
[`ParametricInstance::new`](crate::ParametricInstance::new) and
[`ParametricInstanceBuilder::parameters`](crate::ParametricInstanceBuilder::parameters).
Legacy `ommx.v1.Parameter` rows are still used at protobuf serialization
boundaries, while Python exposes the corresponding SDK class as
`ommx.Parameter`; Rust parsing drains their inline IDs and labels into
`ParameterTable`, and Rust serialization materializes them back from the table.

Related PR: [#967](https://github.com/Jij-Inc/ommx/pull/967).

## рЯІЃ Instance classes for adapter inputs

An adapter input is an OMMX [`Instance`](crate::Instance) value passed to an
adapter. An [`InstanceClass`](crate::InstanceClass) is a set of such values
defined in terms of the variable kinds referenced by the objective and active
constraints, objective degree, regular- and Indicator-constraint relations and
degree bounds, OneHot / SOS1 presence, and optimization sense.

An `InstanceClass` is represented as a finite union of
[`InstanceClassClause`](crate::InstanceClassClause) values. Conditions within a
clause are conjunctive, while clauses are alternatives. This can express, for
example, the union of continuous QP and MILP without incorrectly including
MIQP. Objective, regular-constraint, and Indicator-body bounds use cumulative
[`DegreeBound`](crate::DegreeBound) values.

[`InstanceClass::check_membership`](crate::InstanceClass::check_membership)
derives private, ephemeral facts from the exact instance on every call and
returns structured per-clause mismatches with affected variable or constraint
IDs. If preparation or lowering produces another `Instance` to use as adapter
input, membership must be checked on that exact value. Membership does not
include adapter-specific preconditions or whether the adapter returns an
output successfully.

[`AdditionalCapability`](crate::AdditionalCapability),
[`Instance::required_capabilities`](crate::Instance::required_capabilities),
and [`Instance::reduce_capabilities`](crate::Instance::reduce_capabilities)
form the explicit special-constraint inspection and lowering API. They do not
define an `InstanceClass` or prove membership after lowering.

Instance classes are also independent of `ommx.v2` `required_features`:
serialization features determine whether a reader can reconstruct a wire
payload exactly, while class membership describes the OMMX-defined
mathematical structure of an adapter input.

## вЪЩпЄП Unified error surface

The default error type is [`ommx::Result<T>`](crate::Result) /
[`ommx::Error`](crate::Error), re-exports of `anyhow::Result<T>` and
`anyhow::Error` so downstream crates can propagate with `?` without
taking an `anyhow` dependency themselves. New crate-level fail-site
macros вАФ [`bail!`](crate::bail) / [`error!`](crate::error!) /
[`ensure!`](crate::ensure) вАФ emit a `tracing::error!` event alongside
producing the `anyhow::Error`, so diagnostic context lands in the
configured tracing subscriber rather than being stacked via
`anyhow::Error::context(...)` at the fail site.

The previous discriminant-style error enums (`InstanceError`,
`MpsParseError`, `StateValidationError`, `LogEncodingError`,
`UnknownSampleIDError`, the variants of `QplibParseError`, вА¶) have
been removed вАФ downstream code never matched on their variants in
practice. A handful of typed surfaces are deliberately kept:

- A curated set of **signal types** ([`InfeasibleDetected`](crate::InfeasibleDetected),
  [`BoundError`](crate::BoundError),
  [`DecisionVariableError`](crate::DecisionVariableError),
  [`DuplicatedSampleIDError`](crate::DuplicatedSampleIDError),
  [`SubstitutionError`](crate::SubstitutionError), вА¶) returned typed
  by their entry-point APIs (`Bound::new`, `Sampled::append`,
  `Substitute::*`, вА¶) for callers that recover by discriminant.
- Two narrow-domain parser errors that carry *positional* breadcrumbs
  ([`ParseError`](crate::ParseError),
  [`qplib::QplibParseError`](crate::qplib::QplibParseError)),
  converted to `ommx::Error` at the domain boundary.

Malformed protobuf bytes passed to the public SDK decoders are normalized to
[`ParseError`](crate::ParseError) before entering the semantic parser. The
decoder methods keep returning `ommx::Result`, while callers can reliably
downcast the top-level error to `ParseError` without depending on `prost`; the
standard source chain still exposes the underlying wire-decoding error for
diagnostics.

Artifact image-reference parsing follows the same pattern. [`ImageRef::parse`](crate::artifact::ImageRef::parse)
keeps returning `ommx::Result`, while invalid caller input retains a
downcastable [`ImageRefParseError`](crate::artifact::ImageRefParseError) and its
underlying OCI parser source. If an invalid name and reference pair is already
persisted in the Local Registry, listing retains the owner-level
[`InvalidLocalRegistryImageRef`](crate::artifact::local_registry::InvalidLocalRegistryImageRef)
signal so callers can distinguish corrupted registry state from current input.

Experiment attachment lookup follows the same pattern. Name-based read APIs
keep returning `ommx::Result`, while an absent name retains a downcastable
[`AttachmentNotFound`](crate::experiment::AttachmentNotFound) signal. Dynamic
Experiment and Run handles inspect descriptor metadata for attachment names and
logical media types without reading payload bytes; payload access still
promotes the descriptor to a registry-backed `StoredDescriptor` before reading
the CAS blob.

Model construction keeps its signal owners as well. Duplicate decision-variable
rows retain [`DecisionVariableError`](crate::DecisionVariableError), while
cross-table parameter collisions use
[`AddDecisionVariableError`](crate::AddDecisionVariableError). Public
[`ContentFactorError`](crate::ContentFactorError),
[`OneHotConstraintError`](crate::OneHotConstraintError), and
[`Sos1ConstraintError`](crate::Sos1ConstraintError) signals distinguish model
validation failures through the `ommx::Result` error chain.

Evaluation APIs retain individual downcastable signals for recovery-oriented
control flow: [`MissingStateEntries`](crate::MissingStateEntries),
[`UnknownStateEntries`](crate::UnknownStateEntries),
[`InconsistentDependentValue`](crate::InconsistentDependentValue), and
[`UnverifiableDependentAssertion`](crate::UnverifiableDependentAssertion).
Caller-provided numeric validation reuses
[`DecisionVariableError`](crate::DecisionVariableError); evaluation does not
define an umbrella error type. Direct function and polynomial partial
evaluation also preserve [`CoefficientError`](crate::CoefficientError) in the
error chain instead of replacing it with an untyped message. Coefficient
overflow while normalizing `Instance`-owned dependencies or restoring removed
constraints against stored dependencies and fixed values, non-finite values
computed from a dependency, and untyped special-constraint propagation failures
remain ordinary `ommx::Error` values.

The curated signal-type list and downcast / propagation patterns live
in the [error handling tutorial](crate::doc::tutorial::error_handling).

## вЪЩпЄП Domain types replace `v1_ext`

The proto-generated `v1::Instance` / `v1::Constraint` / `v1::Function` types
are now reserved for wire-format interop. All in-memory operations вАФ QUBO/HUBO

on the domain types ([`Instance`](crate::Instance),
[`Constraint`](crate::Constraint), [`Function`](crate::Function), вА¶) in the
crate root. The `v1_ext` helper module has been removed.

Two new domain traits accompany this shift:

- [`Propagate`](crate::Propagate) performs unit-propagation-style constraint
  reasoning and returns a [`PropagateOutcome`](crate::PropagateOutcome) вАФ
  `Active(T)` (constraint shrunk in place), `Consumed(T)` (fully determined
  by the state, move to removed), or `Transformed { original, new }` (kind
  change, e.g. an indicator constraint promoted to a regular constraint).
  Callers that need provenance bookkeeping append a `Provenance` entry to
  the host's [`ConstraintContextStore`](crate::ConstraintContextStore)
  when they apply the outcome.
- [`Substitute`](crate::Substitute) performs symbolic variable substitution,
  with an acyclic fast path and full cycle detection.

## рЯІЃ Fallible coefficient and function arithmetic

[`Coefficient`](crate::Coefficient) is a finite, non-zero value type again.
Arithmetic that would create NaN or infinity returns
`Err(CoefficientError)`, while cancellation or underflow to zero removes the
term from its polynomial container. [`Function`](crate::Function) and
`PolynomialBase` operations that can violate this invariant therefore return
`Result`; Rust callers migrating operator-heavy code generally need to add `?`
or handle the error explicitly.

The operator surface is also more complete: linear, quadratic, and dynamic
monomials support subtraction across owned and borrowed forms, and Function can
be divided by a Coefficient. Function results normalize back to the smallest
matching representation (`Zero`, `Constant`, `Linear`, `Quadratic`, or
`Polynomial`) after polynomial-level computation. Bulk expression builders can
pre-allocate term storage with `PolynomialBase::with_capacity` and `reserve`.

See the [fallible arithmetic migration](crate::doc::migration_guide#fallible-coefficient-and-function-arithmetic)
for construction and operator examples.

## вЪЩпЄП OMMX Artifact lifecycle and OCI representation

Artifact construction is registry-first in v3. The generic v2
`Builder<Base: ImageBuilder>` split between a local OCI directory and an
archive has been replaced by one lifecycle:

1. [`ArtifactDraft`](crate::artifact::ArtifactDraft) is the mutable SDK object.
   Its `add_*` methods store typed payloads or arbitrary layer bytes in the
   Local Registry immediately and retain registry-backed descriptors.
2. `commit()` seals the root OCI Image Manifest and publishes its digest under
   the requested image reference. `commit_replace()` is the explicit
   last-writer-wins variant; ordinary commit reports a conflicting ref instead
   of silently moving it.
3. The result is an immutable [`LocalArtifact`](crate::artifact::LocalArtifact).
   [`LocalArtifactDyn`](crate::artifact::LocalArtifactDyn) and
   [`LocalRegistryHandle`](crate::artifact::LocalRegistryHandle) provide owned
   handles where borrowing a registry is impractical, and
   [`AsArtifact`](crate::artifact::AsArtifact) lets Artifact-backed domain
   objects share save and push operations.

An OMMX Artifact is represented by an OCI Image Manifest with `artifactType`,
an OCI empty config for generic artifacts, typed layers, optional annotations,
and an optional OCI `subject` for lineage. The deprecated OCI Artifact Manifest
is not used. [`StoredDescriptor`](crate::artifact::local_registry::StoredDescriptor)
ties a descriptor to the Local Registry that owns its bytes, so callers read
payloads through Artifact or registry handles rather than treating a digest as
a filesystem path.

Typed `Instance`, `ParametricInstance`, `Solution`, and `SampleSet` layers are
written with the `ommx.v2` media types. Readers and the public media-type
predicates accept both v1 and v2 layers, so existing v1 artifacts remain
readable. The [Artifact specification](crate::doc::artifact_design) describes
the stable OCI boundary, and the
[Artifact migration guide](crate::doc::migration_guide#rust-sdk-v3-artifact-api-migration-guide)
maps the removed v2 builders to the v3 workflow.

## вЪЩпЄП Artifact archives and remote distribution

The Local Registry is the source of truth for loaded Artifacts. A `.ommx` file
is now a deterministic tar of an OCI Image Layout used for exchange, not an
Artifact opened and mutated in place. Export with
[`LocalArtifact::save`](crate::artifact::LocalArtifact::save); import with
[`LocalArtifactDyn::import_archive`](crate::artifact::LocalArtifactDyn::import_archive)
or [`LocalRegistry::import_oci_archive`](crate::artifact::local_registry::LocalRegistry::import_oci_archive).
OCI Image Layout directories and v2 per-ref directory trees have explicit,
identity-preserving import APIs instead of being consulted as the v3 hot-path
store. Archive export preserves a root manifest's OCI `subject` descriptor but
does not bundle the subject manifest or its ancestry.

With the `remote-artifact` feature, remote pull and push use the OCI
Distribution API directly. Pull imports the remote manifest, config, and layers
into the Local Registry; it does not recursively fetch a manifest referenced by
OCI `subject`. Push sends config and layer blobs before publishing the manifest,
reuses Docker credential configuration, and skips blobs the destination already
has. The archive and remote paths no longer depend on `ocipkg`.

Remote manifest lookup and Local Registry pulls keep returning
[`ommx::Result<T>`](crate::Result), with the OMMX-owned, non-exhaustive
[`RemoteArtifactError`](crate::artifact::RemoteArtifactError) retained as a
downcastable signal in the error chain. Callers can distinguish an absent
manifest from authentication, authorization, transport, and invalid-Artifact
failures without depending on `oci-client` or matching error strings. Each
variant retains the requested [`ImageRef`](crate::artifact::ImageRef) and the
original source chain. Local Registry storage and index failures are not
classified as remote errors.

## вЪЩпЄП `ommx::artifact::ImageRef` replaces `ocipkg::ImageName`

`ImageRef` is the OMMX-owned parsed form of an OCI image reference,
implemented as a thin newtype around
[`oci_spec::distribution::Reference`]. It accepts
`host[:port]/name:tag`, `host[:port]/name@<digest>`, and the combined
`name:tag@<digest>` form on parse, and canonicalises digest references
to `host[:port]/name@<digest>` on Display (tag references keep the
`:` separator). `ImageRef` supersedes the previously re-exported
`ocipkg::ImageName` on every public surface:
[`LocalArtifact::open`](crate::artifact::LocalArtifact::open),
[`ArtifactDraft::new`](crate::artifact::ArtifactDraft::new),
the SQLite Local Registry helpers, and the CLI parse path all now take
[`ImageRef`](crate::artifact::ImageRef). The accessor shape is
`registry()` (the joined `host[:port]` form) plus `name()` /
`reference()` вАФ the v2 split accessors `hostname` / `port` have
been removed since every internal consumer ended up rejoining them
back to `host[:port]` at the call site (callers that genuinely need
just the host portion, e.g. the localhost / 127.* heuristic in
`remote_transport::protocol_for`, parse the joined form inline).
Bare-namespace inputs without an explicit registry
(`library/ubuntu:20.04`, `alpine`) default to `docker.io` via the
standard Docker reference heuristic вАФ the first segment is only
treated as a host when it contains `.` or `:` or equals `localhost`.
The `ommx::ocipkg` re-export is removed.

### v2 compat shim for Docker Hub shorthand

SDK v2 used `ocipkg`, which defaulted bare image names (`alpine`,
`ubuntu:20.04`) to the hostname `registry-1.docker.io`. v3 uses
`oci_spec::distribution::Reference`, whose canonical Docker Hub
hostname is `docker.io` (with `library/` automatically prefixed for
single-segment names). Without intervention, the same image would
land under two distinct SQLite Local Registry keys depending on
which side of the v2 вЖТ v3 boundary produced the string:
`load("alpine")` вЖТ `docker.io/library/alpine`, but a v2 cache
annotated as `registry-1.docker.io/alpine:latest` вЖТ
`registry-1.docker.io/alpine`. [`ImageRef::parse`](crate::artifact::ImageRef::parse)
rewrites the `"registry-1.docker.io/"` prefix to `"docker.io/"` before
delegating to `oci_spec`, collapsing every spelling of a Docker Hub
image onto the same canonical key. The rewrite requires the trailing
slash so adjacent hostnames like `registry-1.docker.io.example/...`
are left alone. `ocipkg`'s legacy `name:algorithm:hex` digest
spelling is *not* preserved вАФ `oci_spec` rejects it вАФ so digest-pinned
v2 annotations must already use the OCI-standard `name@<digest>`
form to round-trip (which is what `ocipkg`'s archive writer emitted
for digest refs in practice).

### `as_path` / `from_path` moved to the legacy module

`ocipkg::ImageName` exposed `as_path()` / `from_path()` for the v2
disk-cache local registry layout (`<root>/<image_name>/<tag>/`). The
v3 SQLite Local Registry stores blobs content-addressed and refs in
SQLite, so per-image directory paths are no longer a v3 concept.
The path-shape helpers have moved off `ImageRef` and are now internal to
the Local Registry implementation; the public v2-path compatibility entry point is
[`LocalRegistry::legacy_ref_path_in`](crate::artifact::local_registry::LocalRegistry::legacy_ref_path_in),
used by `ommx import-legacy` and the CLI's v2-only migration hint.
The previously public `get_image_dir` /
`image_dir` functions and the `ommx image-dir` CLI subcommand are
removed вАФ their return value no longer corresponded to any v3
storage location.

### Accessor behaviour notes

- `reference()` returns the digest when both a tag and a digest are
  set (the OCI `name:tag@<digest>` combined form). OMMX has no code
  path that produces the combined form, so the tag-drop is theoretical
  for SDK-internal use; external callers who manually pass the combined
  form should be aware that round-tripping `ImageRef` through
  `(repository_key, reference)` keeps only the digest.

## рЯІЃ Model transformations and legacy constraint semantics

Integer transformation now includes
[`Instance::unary_encode`](crate::Instance::unary_encode) for narrow finite
ranges alongside [`Instance::log_encode`](crate::Instance::log_encode). Both
encoders use ATol-aware bound normalization, reject ranges that cannot be
represented safely, operate transactionally over multiple variable IDs, and
reject fixed variables before substitution. Unary encoding uses one binary
variable per unit of range width and is guarded by an explicit `max_range`.

[`Instance::into_partial_evaluated`](crate::Instance::into_partial_evaluated)
is the consuming counterpart to
[`Evaluate::partial_evaluate`](crate::Evaluate::partial_evaluate). Use the
borrowed form when the original instance must remain available on error, and
the consuming form when ownership can be transferred to the rewrite.

Legacy v1 `ConstraintHints` remain advisory during deserialization. The v1
readers preserve the referenced regular constraints and context but do not
promote unverified hints into first-class one-hot or SOS1 constraints.

## вЪЩпЄП Experiment sessions and Run lifecycle

OMMX v3 introduces [`Experiment`](crate::experiment::Experiment) as a mutable
session for recording optimization work. An Experiment owns shared attachments
and opens [`Run`](crate::experiment::Run) scopes; each Run owns its parameters,
attachments, trace, solves, and samplings. Non-parameter payloads are written to
the Local Registry as soon as they are logged, while the mutable session retains
registry-backed descriptors until commit.

`Experiment::commit` can run only when no live Run handle exists. The borrowed
API enforces that relationship statically, while dynamic commit checks it at
runtime; an abandoned Run is closed without being recorded. Commit seals the
Experiment config and referenced payloads into one immutable Artifact and
returns a [`SealedExperiment`](crate::experiment::SealedExperiment) for typed
read access. Runs explicitly finish, fail, or record interruption, and sealed
views expose those statuses without reopening the mutable session. Experiments
may use a caller-provided [`ImageRef`](crate::artifact::ImageRef) or an
anonymous local name generated by the registry.

[`ExperimentDyn`](crate::experiment::ExperimentDyn),
[`RunDyn`](crate::experiment::RunDyn), and the other dynamic handles own their
registry relationship and do not expose Rust lifetimes. They support runtimes
and language bindings that cannot retain the borrowed static API while sharing
the same persisted format and invariants.

## вЪЩпЄП Solve and Sampling records

A Run records solver and sampler calls as separate operation kinds. A
[`Solve`](crate::experiment::Solve) owns an input Instance, optional output
Solution, adapter name and options, lifecycle status, and optional adapter
diagnostics. A [`Sampling`](crate::experiment::Sampling) has the parallel
shape with an optional SampleSet output and an independent ID sequence. A
successful sampling remains `finished` even when its SampleSet contains no
feasible sample.

Rust adapters and drivers record operations through `Run::log_finished_solve`,
`Run::log_failed_solve`, `Run::log_finished_sample`, and
`Run::log_failed_sample`, or reserve an ID before a manual workflow and finalize
that ID later. Failed and interrupted attempts remain inspectable with their
input, adapter metadata, and any diagnostics gathered before failure; output is
present only for a successfully decoded result. Sealed and dynamic Run views
expose solves and samplings as distinct collections.

[`AdapterDiagnosticPayload`](crate::experiment::AdapterDiagnosticPayload)
validates the MessagePack-array diagnostics boundary shared by both operation
kinds.

## вЪЩпЄП Experiment attachments, parameters, and traces

[`AttachmentLogger`](crate::experiment::AttachmentLogger) is implemented by
Experiment and Run handles. It logs arbitrary bytes, JSON, files, and typed
OMMX roots into a name-indexed [`AttachmentTable`](crate::experiment::AttachmentTable).
Readers list names, inspect logical media types, decode common OMMX payloads,
read raw bytes, or restore file attachments without exposing registry-internal
descriptors.

File- and reader-backed logging streams into the Local Registry rather than
buffering the complete payload. Attachments may opt into zstd compression;
reserved layer metadata records the storage encoding separately from the
logical media type, so typed and raw readers transparently decompress the
payload. Typed OMMX attachments use v2 layers by default and still read v1
layers, while domain annotations survive the Artifact projection round trip.

Run parameters are intentionally separate from attachments. `log_parameter`
accepts bool, signed integer, float, and string observations, and commit
materializes all Runs into a typed column-oriented MessagePack table. Floating
columns preserve NaN and positive or negative infinity. Each Run may also store
one opaque OTLP protobuf [`Trace`](crate::experiment::Trace), retained through
commit and fork.

## вЪЩпЄП Experiment persistence, recovery, and lineage

Committed Experiments use the dedicated
`application/org.ommx.v1.experiment` OCI `artifactType`, with
`application/org.ommx.v1.experiment.config+json` as the validated Experiment
config schema boundary. They can be opened, imported from an archive, saved,
pushed, or tagged through the same Artifact transport. Earlier v3 alpha
Experiments used the generic Artifact type and must be recreated; this is an
intentional prerelease format break.

Forking a sealed Experiment creates a new mutable child with the parent's
attachments, Runs, operation records, parameters, and next Run ID. The child
manifest records the parent manifest as its OCI `subject`, so lineage is
explicit while content-addressed payload blobs remain shared.

By default, closing a Run makes a best-effort attempt to publish a rolling draft
checkpoint under an internal ref derived from the requested Experiment image
name. Dynamic callers explicitly publish abnormal terminal state through
`ExperimentDyn::commit_failed_checkpoint` or
`commit_interrupted_checkpoint`. Successful commit makes a best-effort attempt
to remove the checkpoint, and
[`ExperimentDyn::restore_from_checkpoint`](crate::experiment::ExperimentDyn::restore_from_checkpoint)
resumes the latest closed-Run state. [`AutosavePolicy`](crate::experiment::AutosavePolicy)
can keep every-Run behavior, batch by Run count, rate-limit by time, or disable
rolling draft checkpoints; it never suppresses explicit failed or interrupted
checkpoints.

Failed and interrupted Experiment and Run records can persist a concise
lifecycle reason in the Experiment config. Use
[`Run::finish_failed_with_reason`](crate::experiment::Run::finish_failed_with_reason),
[`Run::finish_interrupted_with_reason`](crate::experiment::Run::finish_interrupted_with_reason),
and their runtime-owned [`RunDyn`](crate::experiment::RunDyn) counterparts to
record one explicitly. [`SealedExperiment::lifecycle_reason`](crate::experiment::SealedExperiment::lifecycle_reason)
and [`SealedRun::lifecycle_reason`](crate::experiment::SealedRun::lifecycle_reason)
read the persisted value. Existing no-reason methods and older configs remain
supported. Lifecycle reasons are not adapter diagnostics and should not contain
secrets, tracebacks, local variables, or environment values.

[`ExperimentLifecycle`](crate::experiment::ExperimentLifecycle) and
[`RunLifecycle`](crate::experiment::RunLifecycle) represent the complete Rust
lifecycle as tagged sum types. Only their `Failed` and `Interrupted` variants
can contain a reason. `SealedExperiment::lifecycle` and `SealedRun::lifecycle`
return those values, while the existing `status` and `lifecycle_reason`
accessors are derived views. The serialized config keeps the existing
top-level `status` and optional `outcome.reason` JSON shape.

Rust callers can compose those transitions with
[`Experiment::scoped`](crate::experiment::Experiment::scoped) and
[`Experiment::scoped_run`](crate::experiment::Experiment::scoped_run):

```rust,ignore
let experiment = Experiment::scoped(Name::Anonymous, |experiment| {
    experiment.scoped_run(|run| {
        run.log_parameter("seed", 1_i64)?;
        Ok(())
    })?;
    Ok(())
})?;
```

Successful callbacks commit or finish, returned errors preserve the original
caller error after a best-effort failed transition, and panic unwind records an
interrupted Run and Experiment checkpoint. Runs borrow the scoped Experiment,
so they cannot escape its callback. Runs opened directly inside a scoped
Experiment inherit its interrupted-on-drop behavior, preserving partial state
even when callers do not use `scoped_run`. Ordinary borrowed Runs retain their
existing abandon-on-drop behavior unless
[`Run::interrupt_on_drop`](crate::experiment::Run::interrupt_on_drop) is selected.
Rust callers that cannot use an Experiment callback can use the existing
explicit `commit*` and `finish*` methods.
[`ExperimentDyn::interrupt_on_drop`](crate::experiment::ExperimentDyn::interrupt_on_drop)
and [`RunDyn::interrupt_on_drop`](crate::experiment::RunDyn::interrupt_on_drop)
add an opt-in, best-effort interrupted fallback to those existing handles.

Caller annotations are stored on the committed Experiment manifest.
`LocalRegistry::list_experiments` and `list_experiment_checkpoints` expose those
annotations together with status, Run/Solve/Sampling counts, and the validated
Experiment config for catalog and recovery workflows.

## вЪЩпЄП Local Registry lifecycle and inspection

The v3 Local Registry replaces the v2 per-image OCI directory tree with a
SQLite ref index and an immutable filesystem content-addressed store. Mutable
refs point to manifest digests; stored manifests, configs, and layers remain
immutable blobs, while an Experiment lineage link is an OCI `subject`
descriptor on its manifest. Store, seal, publish, and replace are separate
operations, and registry-backed descriptors prevent callers from accidentally
reading a digest through the wrong registry. Legacy v2 layouts are imported
explicitly and preserve their original manifest identity.

The Local Registry exposes typed Artifact, Experiment, and checkpoint listings
with cached Manifest and Experiment config metadata. Best-effort listings
repair or skip invalid per-ref entries and report structured warnings, while
strict mode aborts at the first individual failure. Opening a version 1 Local
Registry migrates it in place to the current schema.

Mutable refs can be removed without immediately deleting their immutable
content-addressed blobs, restored from the removed Manifest digest, or pruned
under anonymous and age-based retention policies. Garbage collection remains a
separate reachability-based operation rooted at live refs, Experiment
checkpoints, protected digests, and recursively traversed config, layer, and
subject descriptors. Report-first cleanup is non-destructive unless deletion is
requested explicitly.

[`ArtifactRefRecord::referenced_blob_size`](crate::artifact::local_registry::ArtifactRefRecord::referenced_blob_size)
and the `ommx size` CLI report the logical size referenced by each Artifact or
Experiment ref: root manifest JSON, config, and unique layers. OCI `subject`
ancestry is intentionally excluded. Shared blobs are counted once per ref, so
these values are not additive physical Local Registry usage.

## вЪЩпЄП Typed PyO3 bridge for independent extension modules

The companion [`ommx-pyo3-bridge`](https://docs.rs/ommx-pyo3-bridge) crate
lets an independently built PyO3 module return Rust OMMX values as the
canonical classes from the installed Python `ommx` package. Its `PyFunction`,
`PyConstraint`, `PyDecisionVariable`, and `PyInstance` wrappers implement both
PyO3's `IntoPyObject` and `pyo3-stub-gen`'s `PyStubType`, so generated stubs use
`ommx.Function`, `ommx.Constraint`, `ommx.DecisionVariable`, and
`ommx.Instance` without per-function type overrides.

The bridge crosses the shared-library boundary with protobuf byte payloads
instead of Rust values or private PyO3 wrapper types. Detached constraints
carry their complete [`ConstraintContext`](crate::ConstraintContext), detached
decision variables carry their table-owned [`VariableID`](crate::VariableID)
and [`ModelingLabel`](crate::ModelingLabel), and instances use their
owner-complete v2 root serialization.

The installed Python SDK owns the binding-private production receivers that
reconstruct its canonical classes; they are not user-facing component
serialization APIs.

This release isolates its endpoint signatures and payload interpretations as
internal bridge protocol v0. Protocol v0 is not a new persistence format, and
the bridge does not perform protocol negotiation or guarantee compatibility
between arbitrary Rust and Python OMMX release pairs. Callers must install a
Python OMMX release exposing the exact v0 capabilities expected by the bridge;
a missing capability raises an `ImportError` that names it.

The signature and payload meaning of an exposed v0 capability stay fixed;
incompatible changes must use a new protocol version rather than reinterpret
v0. Supporting or retiring a protocol remains a separate Python SDK lifecycle
decision, and the bridge performs no version fallback.

Related PR: [#1106](https://github.com/Jij-Inc/ommx/pull/1106).

## вЪЩпЄП Other notable changes

- [`Instance::logical_memory_profile`](crate::Instance::logical_memory_profile)
  reports structural memory use across v3 constraint, table, label, and context
  owners. Derive-based coverage keeps the report aligned with their normalized
  representations.
- `ommx::doc` is now the entry point on docs.rs for long-form prose
  (this page, the [migration guide](crate::doc::migration_guide), and the
  [tutorial](crate::doc::tutorial)).
- [`CURRENT_FORMAT_VERSION`](crate::CURRENT_FORMAT_VERSION) replaces v2's
  `ACCEPTED_FORMAT_VERSION` name. Top-level v1 writers stamp this value and
  readers reject data produced with a higher semantic format version.
- The workspace minimum supported Rust version is **1.89**.
- Official Rust SDK releases include prebuilt `ommx` CLI archives and metadata
  for installation with `cargo binstall ommx`.
